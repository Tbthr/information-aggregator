# 信息聚合器 v1 总体规划：统一策略框架 + X 深化 + 日报主入口

## Summary

本轮规划按“单一大版本、内部阶段实施”推进，目标不是继续扩充采集能力，而是把项目从“聚合工具”升级为“个人注意力管理系统”。

v1 的产品定义如下：

- 目标用户：先做自己用，允许工程化配置，但默认体验要顺滑
- 主入口：Web 首页默认进入“日报”
- 个性化方式：AI 优先，规则与 prompt 配置作为兜底和校正机制
- 核心能力：按来源类型采取不同处理策略，显式区分“过滤模式”和“阅读辅助模式”
- 范围覆盖：统一策略框架、X 场景深化、日报/周报、Pack/来源双维度浏览，一次纳入同一方案，但按阶段落地

推荐实施顺序：

1. 建立来源策略模型与 AI 判断链路
2. 先打通 X 场景的差异化体验
3. 以日报为主入口重构 Web 信息组织
4. 补齐周报与长期沉淀视图

## Key Changes

### 1. 产品模型与策略层

新增一层 `Source Policy` / `Content Treatment Policy`，作为 Pack 与 adapter 之间的产品语义层，但不改变现有 deterministic pipeline 原则；AI 仍是可选增强层。

需要定义的策略对象：

- `sourceMode`
  - `assist_only`：不过滤，只做阅读辅助
  - `filter_then_assist`：先过滤，再做阅读辅助
  - `filter_only`：只做筛选，不做增强
- `sourceIntent`
  - `trusted_curated`
  - `semi_curated`
  - `open_stream`
  - `reference_archive`
- `aiTasks`
  - `relevance`
  - `value`
  - `actionability`
  - `one_line_summary`
  - `reading_hint`
- `uiSurface`
  - `daily`
  - `weekly`
  - `pack`
  - `source`

默认映射规则：

- `x_bookmarks` → `assist_only`
- `x_home` → `filter_then_assist`
- `x_list` → `filter_then_assist`
- `x_likes` → `assist_only`
- `x_user_tweets` / `x_search` → `filter_then_assist`
- 高质量 RSS / JSON Feed → 默认 `filter_then_assist`
- 纯收藏型来源后续允许切到 `assist_only`

这层配置应优先挂在 pack/source 配置中，避免把行为散落到 view 和 prompt 中。

### 2. AI 判断与解释链路

新增内容判断的统一输出结构，供 API、Web、日报、周报复用。v1 不追求复杂学习系统，只做“可解释的 AI 判断结果”。

建议新增标准判断字段：

- `relevanceLabel`：`high | medium | low`
- `valueLabel`：`high | medium | low`
- `actionabilityLabel`：`high | medium | low`
- `keepDecision`：`keep | maybe | drop`
- `keepReason`：一句话说明为什么保留/过滤
- `readerBenefit`：读它的收益
- `readingHint`：建议怎么读
- `confidence`：低/中/高

判断规则：

- `assist_only` 来源永不因 AI 被丢弃，只补充解释与建议
- `filter_then_assist` 来源先给出 `keepDecision`，仅 `keep/maybe` 进入日报主链路
- `daily` 使用 `keepDecision + relevance + value` 排序
- `weekly` 使用“重复出现主题 + 持续价值”聚合，而不是简单取 7 天高分内容

### 3. X 场景深化

X 是 v1 的主战场，但不做 X 特化分叉架构，而是在统一策略层上做第一套完整落地。

X 各来源行为：

- `x_bookmarks`
  - 默认全保留
  - 每条生成一句话总结、对我的启发、建议阅读动作
  - 支持按主题/来源账号二次聚合
- `x_home`
  - 默认强过滤
  - 重点判断“是否和我有关”“是否值得现在读”
  - 输出应突出过滤理由
- `x_list`
  - 作为半结构化高质量流
  - 允许按 list 定义独立 prompt 或主题偏好
- `x_likes`
  - 视作弱书签
  - 默认不过滤，但排序低于 bookmarks
- `x_search` / `x_user_tweets`
  - 作为任务型来源
  - 优先进入来源页和 Pack 页，不默认进入日报头部

### 4. Web 与消费入口

Web v1 采用“日报首页 + 双维导航”结构。

页面结构：

- 首页：日报
  - 今日最值得看
  - 可快速解释“为什么是这些”
  - 拆分为 `必须看 / 可略读 / 值得收藏`
- Pack 页
  - 按主题消费
  - 展示该 Pack 的策略摘要、来源构成、今日/本周代表内容
- 来源页
  - 按 source 查看
  - 强调来源类型、当前策略、过滤比例、推荐逻辑
- 周报页
  - 展示一周高价值主题、反复出现信号、值得跟踪的问题

导航与筛选原则：

- 首屏不暴露太多工程参数
- Pack 与来源是一级切换维度
- 日报/周报是一级时间消费视图
- “为什么推荐/为什么过滤”必须在卡片层可见，不藏在详情页里

### 5. 配置、API 与类型边界

v1 需要补的公共接口，不要求一次把底层 schema 做复杂，但必须统一语义。

配置新增项：

- Pack 或 source 级别支持策略定义
- 来源级 prompt 覆盖
- 来源级默认模式覆盖
- 用户关注主题 / 排除主题的最小配置入口
- 日报与周报的入选阈值配置

API 需要扩展的能力：

- `items` 响应增加策略判断字段与解释字段
- 增加“按来源聚合”的查询能力
- 增加日报接口
- 增加周报接口
- `packs` 响应增加策略摘要和来源分布信息

视图与类型需要新增的核心对象：

- `SourcePolicy`
- `ContentDecision`
- `DailyDigestSection`
- `WeeklyThemeSummary`
- `SourceViewSummary`

默认假设：

- 继续沿用现有 archive 与 query 体系
- 不重做 adapter 层
- 不在 v1 引入用户账户、多用户设置或在线写回反馈系统
- 用户偏好先由本地配置文件承载

## Delivery Plan

### Phase 1：策略层打底

目标：建立统一来源策略框架，让不同来源的处理模式可配置、可解释、可被 API 复用。

交付标准：

- 已有 source type 都能映射到明确策略模式
- `assist_only` 与 `filter_then_assist` 两条主链路明确
- AI 输出结构统一
- API 可返回“是否保留”和“为什么”

### Phase 2：X 深化落地

目标：把 X 作为第一类完整样板，验证统一框架是否成立。

交付标准：

- `x_bookmarks` 体验从“筛选”转为“阅读辅助”
- `x_home` / `x_list` 支持过滤理由与阅读价值判断
- 来源页能体现 X 各来源的差异化处理
- prompt 可按来源覆写

### Phase 3：日报主入口

目标：让 Web 从“查询工具”升级为“日常消费入口”。

交付标准：

- 首页默认日报
- Pack/来源双维导航可切换
- 卡片层可见推荐解释
- 日报结果明显少于普通流，形成“少而精”体验

### Phase 4：周报与长期沉淀

目标：从“当下筛选”延伸到“持续洞察”。

交付标准：

- 周报可聚合重复主题与长期信号
- 区分“今天值得读”和“这周值得理解”
- 可从周报跳回相关来源、Pack、单条内容

## Test Plan

需要覆盖的验证场景：

- `x_bookmarks` 内容不会被过滤，但会生成阅读建议
- `x_home` 相同内容在过滤链路中可得到 `keep/maybe/drop`
- `assist_only` 与 `filter_then_assist` 输出结构一致，但行为不同
- 来源级 prompt 覆盖不会破坏默认策略
- 日报首页能稳定只展示保留内容，不混入 `drop`
- 周报聚合不只是时间窗口拼接，而能形成主题总结
- Pack 视图与来源视图对同一条内容展示一致的解释字段
- 无 AI 配置时系统仍能运行，回退到 deterministic 排序与基础摘要
- 新增策略字段后，旧 pack 配置仍可加载并使用默认值

## Assumptions

- 本轮是产品与实施规划，不涉及立即编码
- v1 先服务单用户、本地优先，不引入账户系统
- AI 判断以“辅助决策、可解释”为目标，不做全自动黑盒裁决
- “一次性都做好”理解为纳入同一总版本目标，不是单次提交同时完成所有开发
- 若后续开始实施，建议先输出一份设计文档，再拆为 3-4 个独立执行计划：策略层、X 场景、日报 Web、周报沉淀
